1. 基本知识

经典的设计模式有23个，这是基于java后端写的，前端不是都常用。

UML

1.面向对象的特征

1.抽象

隐藏方法的实现，只关心使用，不关心具体实现。在ts中，使用interface

2.继承

  1. 继承主要是实现代码逻辑的复用，或者说是属性和方法的复用

  2. 继承也会导致类的层次过深，类和类之间出现了耦合，如果修改父类，子类也会跟着变

3.封装

仅暴露必要的接口，把内部的属性隐藏起来，不让外部直接访问。

  1. 提高类的易用性

  2. 减少耦合

实现需要利用到ts的修饰符。

public:公有修饰符，可以在类内或者类外使用public修饰的属性或者行为，默认修饰符

protected:受保护的修饰符，可以本类和子类中使用protected修饰的属性和行为

private :私有修饰符，只可以在类内使用private修饰的属性和行为

4.多态

  1. 子类可以替代父类

  2. 子类可以重写父类中的方法。

2.设计原则 SOLID

1.S单一职责原则

Single responsibility principle

  • —个类或者模块只负责完成—个职责,如果功能系特别复杂就进行拆分单一职责可以降低类的复杂性，提高代码可读性、可维护性

  • 拆分不能过度，如果拆分过度会损失内聚性和维护性

2.O开闭原则

OpenClosed Principle

  • 对扩展开放，对修改封闭

  • 增加需求时，扩展新代码，而非修改已有代码

so原则体现：

  • 单一职责原则：每个then中的逻辑只做好一件事

  • 开放封闭原则（对扩展开放，对修改封闭）：如果新增需求，扩展then

3.L里式替换原则

Liskov Substitution Principle

  • 子类能覆盖父类

  • 父类能出现的地方子类就能出现

子类必须遵循父类的原则，里式替换原则是开闭原则实现的基础。

4.I 接口隔离原则

接口尽量细化，不应该依赖不需要的接口。

5.依赖倒置原则

  • 面向接口编程，依赖于抽象而不依赖于具体

  • 使用方只关注接口而不关注具体类的实现

(例如，找女朋友，现有一个抽象标准，而不是现有一个具体的人（具体的实现）)

3.合成复用原则

少用继承，而多用组合或者聚合。

例如人可以是厨师也可以不是厨师，这是一种关联关系，而使用继承则表示每个人都是厨师。

泛化：老师是数学老师的泛化

依赖： 数学老师和黑板（黑板可有可无） 如果一个类是当前类的方法和和属性，或者是局部变量就是依赖关系。

关联：单向关联和双向关联

聚合：学生和数学老师聚合在班级中

组合：心脏和学生是组合关系

设计模式可以分为创建型，结构型，行为型

1.创建型设计模式

1.简单工厂模式

又叫静态工厂方法（因为工厂类定义了一个静态方法），可以根据参数的不同返回一个类的实例。简单工厂模式专门定义一个类来负责创建其他类的实例，使一个类的实例化延迟到了工厂类。被创建的对象通常都具有相同的父类（工厂创建的所有具体产品应该是一个大类）。

优点

  1. 使对象的创建和使用分离，减少了耦合，客户端不需要关系怎么创建，只需要知道怎么使用。

缺点

  1. 扩展性差，不够灵活（新增加一个产品类，需要修改工厂类方法的判断，增加一个具体产品类）

例子：

  • JQuery的$()就是一个工厂函数，它根据传入参数的不同创建元素或者去寻找上下文中的元素，创建成相应的jQuery对象。

  • React.createElement 方法，根据传入的type，创建对应的虚拟dom。

  • vue 的异步组件

1.通过类实例化对象创建

js

2.创建一个新对对象，包装增强属性和功能实现

2.工厂模式

简单工厂模式每当有新产品加入的时候，必须要修改工厂类，违反了开闭原则，因此工厂模式将简单工厂模式中的工厂变成一个抽象接口。不再负责产品的创建，而是交给子类（具体工厂类）。将产品的实例化操作延迟到工厂子类去完成。 先创建具体的工厂，然后具体的工厂来创建产品

相比较于简单工厂模式，不需要修改工厂逻辑，但是需要新增加一个工厂。

  • 什么不直接修改简单工厂？ 因为如果直接修改了简单工厂违反了开闭原则。举一个例子：一个生产拖拉机的工厂，决定生产口罩，肯定不会再原有的生产线上修改，而是新建立一个工厂，这也说明，扩展优于修改。

应用场景react.createFactroy

tips：安全模式类

当构造函数没有使用new关键字调用时不仅不能创建实例，还会声明全局变量。通过这种方式可以保证无论使用new关键字还是直接调用构造函数，都可以达到预期目的。

tips：不是所有函数都可以当做构造函数

箭头函数， 函数的简写形式不能当作构造函数。

Not every JavaScript function is constructable | Stefan Judis Web Development

3.抽象工厂模式

工厂模式要求所有产品都是同一个“大类”（产品实现同一接口或者继承了同一的抽象类。）抽象工厂模式不但能生产具体某一大类的产品，还可以生产其他大类产品。打破了工厂和抽象产品一对一的关系。

  • 新增一个产品体系，必须修改抽象接口，具体工厂的实现，违反了开闭原则。具体在代码应用取决于开发者。

  • 抽象工厂模式与工厂方法模式最大的区别在于：工厂方法模式针对的是一个产品体系（种类）；抽象工厂模式则需要面对多个产品种类，一个工厂负责可以多个不同种类产品的产品创建。

  • 抽象工厂模式是工厂模式的一个扩展，当产品体系只有一个的时候，就变成了工厂模式。

tips如何实现抽象类

在类的方法中手动抛出错误来模拟抽象类。

tips 如何实现私有属性

通过闭包实现，因为函数中的变量对外不可访问。

4.单例模式

一个类只有一个实例，并提供一个访问它的全局访问点。

相较于java，由于js是单线程的，因此创建单例很简单，但是java是支持多线程的，创建单例需要考虑锁死线程。

优点：

  • 只会实例化一次。简化了代码的调试和维护

  • 划分命名空间，减少全局变量

缺点：

  • 由于单例模式提供的是一种单点访问，所以它有可能导致模块间的强耦合 从而不利于单元测试。无法单独测试一个调用了单例的方法的类，而只能把它与那个单例作为一个单元一起测试。（例如，登录框实现单例模式，在单元测试的时候需要测试每一个调用了单例方法的地方。而不是仅仅测试登录框。）

使用场景

  • vuex 和 redux中的store

  • 登录框，在所有页面都可以调用登录。

5.原型模式

原型模式，就是创建一个共享的原型（js本身就是通过原型实现继承），通过拷贝这个原型来创建新的类，用于创建重复的对象，带来性能上的提升。

创建基类时，简单差异化的属性放在构造函数中，消耗资源或者相同功能的放在原型中。&#x20;

js本身是基于原型链实现对象之间的继承，这种继承是对属性或方法的共享，而类似java语言实现原型模式是对属性和方法的复制。

6.建造者模式

工厂模式关注的是创建实例的结果，而建造者模式更关注的是建造对象的整个过程。创建的对象是一个复合对象。 在js中较少使用。

将多个类的实例作为建造类的属性。

2.结构设计模型

1.外观模式

外观模式是指为一组复杂的子系统接口提供一个更高级的统一接口，通过这个接口使得对子系统接口的访问更容易。

外观模式核心在于对其他接口的二次封装，可以隐蔽其复杂性，简化其使用。

应用

  • js对底层结构兼容性做统一封装是一种外观模式。

  • react 核心 分为 react-dom 和react-native，也是外观模式的一种体现。

注意区分-工厂模式-外观模式：

  • 工厂模式核心是对创建对象的逻辑进行封装。

  • 外观模式核心是对不同的接口进行封装。

2. 适配器模式

又称包装器模式，将一个类（对象）的接口（方法、属性）转化为用户需要的另一个接口，解决类（对象）之间接口不兼容的问题。&#x20;

使用场景：

  1. 如果你想要使用一个已经存在的对象，但是它的接口不满足需求，那么可以使用适配器模式，把已有的实现转换成你需要的接口；（例如，后端传给前端的是一个对象，前端需要的是数组，可以使用适配器模式将数组转换为对象。）

  2. 如果你想创建一个可以复用的对象，而且确定需要和一些不兼容的对象一起工作，这种情况可以使用适配器模式，然后需要什么就适配什么。

应用：

  • Vue 的计算属性（通过计算Data，转换成我们想要的数据。）

  • 适配器传参。将接收的参数转换成对象。

  • axios，adapters 适配器方法 两种xhr（web）、http（node）

适配新的接口

适配器和外观模式的区别：

  • 外观模式是提供给了一个更高级的统一接口

  • 适配器模式是将一个接口转化为另一个接口

3.代理模式

react 的render props 使用了代理模式，react router使用的代理模式。

由于一个对象不能直接引用另外一个对象，所以需要通过代理对象在者两个对象中起到中介的作用。

代理模式能将代理对象与被调用对象分离，降低了系统的耦合度。代理模式在客户端和目标对象之间起到一个中介作用，这样可以起到保护目标对象的作用。

应用：

  • dom事件委托

    如果有许多元素都有一个共同的事件，那么就不必为每个元素都分配一个事件处理程序 —— 而是将单个处理程序放在它们的共同祖先上。

    优点：

    • 减少内存的使用（减少函数的使用）

    • 可以监听动态元素

  • JSONP

    通过script标签可以跨域，在前端写一个全局函数，而在后端进行全局函数的调用。script标签就是那个代理。

  • es6 proxy实现vue3的响应式

注意区分适配器模式，装饰器模式，代理模式；

  • 适配器模式提供不同的新接口，通常用作转换的兼容处理

  • 代理模式提供一模一样的新接口，对行为进行拦截

  • 装饰器模式直接访问原接口，直接对原接口进行功能上的增强

4. 装饰器模式

Rreact HOC, 传入一个组件，返回一个新的组件。利用了这个设计模式。react-redux中的connect( state, action )( component )也是使用装饰器模式。

做到关注点分离，缺点：层层嵌套

向一个现有的对象添加新的属性或者方法，同时又不改变其结构，是作为对现有类的一个包装。

装饰器不需要了解原有对象的功能，原有对象的方法能正常使用，适配器需要了解原有方法的使用，

装饰器是实现 AOP（面向切面）编程的一种重要方式。

5.桥接模式

  • 将抽象部分与它的实现部分分离（解耦），使他们可以独立的变化。

  • 通过关联的关系，而不是继承的关系，减少系统类的个数。

传统⽅式就直接通过继承，就需要特别多的类，品牌2，三个功能3，就是6个类了，如果后续再增加品牌就更多了，类数⽬将会激增，即所谓的类爆炸。

而通过桥接将继承改为关联，将一个类变成一个属性。在属性特别多的时候，效果越明显

6.组合模式

  1. 组合模式是一种结构型设计模式，将对象组合成树状结构以表示”整体-部分”的层次结构。

  2. 组件和叶子对象具有统一的功能和接口。 这样使用组合体和分别使用对象是一样的效果。在树的构造最终完成之后，只需要通过请求树的最顶层对象，便能对整棵树做统一一致的操作。

应用：

create.element

表单组件的设计，Form ，ITEM, ELEMENT

7.享元模式&#x20;

flyWeight 享元，轻量化。 (享:共享) 性能优化.

享元模式是一种优化程序性能的模式, 将状态划分为内部状态(不变或很少变化的属性或者方法)和外部状态(经常变化的属性或者方法). 这样可以减少实例对象的个数,从而优化代码.

类似于共享连接池

应用场景：

  1. 有大量相似的对象, 占用了大量内存

  2. 对象中大部分状态可以抽离为外部状态

**实际例子: **

  1. 页面渲染相同的元素

  2. 原型链的方法

举例:

某商家有 50 种男款衣服和 50 种女款衣服, 需要给模特拍照

方案一: 造 50 个塑料男模和 50 个塑料女模, 让他们穿上展示，很显然，需要创建100个对象，性能低下

方案二: 造 1 个塑料男模特 1 个塑料女模特, 分别试穿 50 款内衣。

将状态（性别，衣服）分为内部状态（较少改变的状态），外部状态（经常改变的状态）

优化：

&#x20;1.new 不同的实例对象应该通过通过工厂方法创建（享元工厂），而不是直接new一个对象

&#x20;2.外部状态添加不明显。添加一个状态管理器，专门添加状态（内部状态和外部状态）。

https://www.bilibili.com/video/BV1UL4y1h7W9?p=28 （优化没看懂，感觉多此一举）

3.行为设计模式

1.策略模式

使用场景:一个问题匹配多个解决方案,而且解决方案不确定会不会修改.如果使用if-else或者case会违反开闭原则,不易扩展.

  • 将定义的一组算法封装起来，使其相互之间可以替换。封装的算法具有一定的独立性，不会随着客户端变化而变化

  • 避免大量的swith case if

应用:例如登录框校验策略

将计算薪资的方法提取出来

由于js不是规定一定要面向对象编程.

2.状态模式

在状态模式,将一个对象的状态抽象出来,当其状态状态改变时行为也会发生改变。

和策略模式一样,也是一种减少if-else语句的方法,另外还有一种:卫语句(判断完直接return)的方法.

在js实现和策略模式类似.

状态模式和策略模式的区别：

  • 状态模式重点在各状态之间的切换，从而做不同的事情；

  • 而策略模式更侧重于根据具体情况选择策略，并不涉及切换。

例如：选择支付方式，是策略模式。 而状态可以切换（困了> 睡觉，饿了>吃饭）

3. 责任链模式

使多个对象都有机会处理请求，从而避免请求的发送者和接受者之间的耦合关系，将这些对象连成一条链，并沿着这条链传递该请求，直到有一个对象处理它为止.

  • JS 中的事件冒泡

  • 作用域链

  • 原型链

  • express 中间件

例如:毕业时,需要导师审批、辅导员审批、宿管审批、图书馆审批,这是一个处理链,任意一个环节都可以处理.

4. 观察者模式

当对象间存在一对多关系时，则使用观察者模式（Observer Pattern）。比如，当一个对象被修改时，则会自动通知依赖它的对象。观察者模式属于行为型模式。

  • 在被观察者发生变化时候，调用观察者的更新方法

该模式由观察者与被观察者组成.

观察者需要实现&#x20;

1.标识符,标识观察者的身份.

2.当时间发生时候的回调函数.

被观察者

1.存储自己状态的属性

2.一个存储观察者的队列

3.修改状态的方法

4.添加监视者的方法 比如dom事件addeventlistener(‘标识的事件名’,事件回调函数)

5.删除监视者的方法 removeeventlistener(‘标识的事件名’)

5.发布订阅

在发布订阅模式里，发布者并不会直接通知订阅者, 而是通过第三方,也就是在消息队列里面.

  • 观察者模式只有两个角色,观察者(observe)和被观察者(subject),发布订阅的模式有三方角色,发布者(出版方),订阅者,经纪人(broker)

  • 观察者和被观察者，是松耦合的关系, 发布者和订阅者，则完全不存在耦合

发布订阅只需要一个构造函数,创建一个第三方broker 经纪人的身份, broker构造函数应该包含

1:属性：消息队列 &#x20;
&#x20; {
&#x20; 故事会：[打电话通知小明,发微信给小红，送货上门给小李]
&#x20; 读者：[送货上门]
&#x20; }
2.添加方法:消息队列中添加内容分
3.删除方法：删除消息队列中的内容
4.触发消息队列中的内容

发布者只需要告诉Broker，我需要发布消息tigger(type)
订阅者只需要告诉Broker,我订阅的是type消息，我先告诉你当消息来了你broker要怎么做。
